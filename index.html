Version control

Git — Basics

git status
Check what changed and whether your branch is clean.
git add -A
Stage everything: new, changed, and deleted files.
git commit -m "your message"
Create a local snapshot with a useful message.
git push
Send local commits to GitHub.
git pull
Bring remote changes into your local repo.
git diff
See exactly what changed before committing.
Recovery / hygiene

Git — Branches & Fixes

git branch
Check which branch you're on. You usually want * main.
git checkout main
Switch to the main branch.
git pull origin main --rebase
Best fix when push is rejected because remote moved ahead.
git rebase --continue
Continue after fixing a rebase conflict.
git remote prune origin
Clean up deleted remote branches from your local list.
git restore index.html
Discard uncommitted changes in one file.
Container setup

Docker — Setup & Run

Install Docker Desktop
Docker Desktop is required on Windows to run Docker containers.
Open Docker Desktop
Wait until Docker is fully running before typing Docker CLI commands.
docker --version
Check Docker is installed and available in terminal.
docker run hello-world
Quick test to confirm Docker can run a container.
Dockerfile needed
A build command only works if the current folder contains a valid Dockerfile.
FROM nginx:alpine
Use nginx as a lightweight static site server.
COPY index.html /usr/share/nginx/html/index.html
Copy your site file into nginx’s web root.
EXPOSE 80
Document the container port used by nginx.
Commands

Docker — Commands

docker build -t cheatsheet-site .
Build an image from the current folder.
docker run -d -p 8081:80 --name cheatsheet-container cheatsheet-site
Run the container in the background and map localhost:8081 to port 80.
docker ps
See running containers.
docker stop cheatsheet-container
Stop the running container.
docker start cheatsheet-container
Restart the old container. This does not rebuild new file changes.
docker rm -f cheatsheet-container
Delete existing container before recreating it.
Mental model

Docker — What It Actually Means

Image = saved snapshot
The built version of your app at a moment in time.
Container = running copy
A live instance running from that image.
docker start = reuse old version
Starting an old container does not pull in your new edits.
Edit file → rebuild → rerun
That’s why Docker feels slower than npm for normal editing.
Local dev

npm — Setup & Run

npm init -y
Create a default package.json in the current folder.
npm install live-server --save-dev
Install live-server as a local dev dependency.
"start": "live-server"
Add this inside the scripts section in package.json.
npm start
Run the local dev server using the start script.
Ctrl + C
Stop the running server in terminal.
npx live-server
Quick fallback if you want to run it without setting up a start script first.
Package file

npm — package.json Example

{ "scripts": { "start": "live-server" } }
Minimal version needed for npm start to work here.
npm run
See which scripts currently exist in package.json.
Missing script: "start"
This means package.json exists, but the start script does not.
Decision making

npm vs Docker

npm = local development
Fast, simple, instant refresh. Best while building.
Docker = container testing
Closer to a clean deployment environment. Best for verification.
Use npm while building
Edit → save → see changes immediately.
Use Docker to verify
Rebuild and run a fresh container to verify the site still works cleanly.

Tips to Remember

Quotes matter
Folders with spaces need quotes, for example cd "Claude Docs".
The dot means current folder
docker build -t app . and git clone URL . both use the current folder.
Push rejected? pull first
Usually git pull origin main --rebase then git push.
Docker failing? check Docker Desktop first
Don’t debug your files before confirming Docker is actually running.
npm start needs package.json
If package.json is missing, npm has nothing to run.
node_modules is local junk, not source code
Keep it out of Git with .gitignore.
