Git — Basics
git status Check what changed and whether your branch is ahead / behind.
git add -A Stage everything including new, changed, and deleted files.
git commit -m "your message" Create a local snapshot with a meaningful message.
git push Send local commits to GitHub.
git pull Bring remote changes into your local repo.
git diff See exactly what changed before committing.
git log --oneline Compact list of recent commits.
Git — Branches & Fixes
git branch Check which branch you’re on. You usually want * main.
git checkout main Switch to the main branch.
git pull origin main --rebase Best fix when push is rejected because remote is ahead.
git rebase --continue Continue after fixing a rebase conflict.
git remote prune origin Clean up deleted remote branches from your local list.
git reset --hard origin/main Force your local files to match GitHub exactly. Use with care.
Docker — Setup & Run
Docker commands will fail unless Docker Desktop is open and running.
Install Docker Desktop Docker Desktop is required on Windows for local Docker containers.
Open Docker Desktop Wait until Docker is fully running before using Docker CLI commands.
docker --version Check Docker is installed and available in terminal.
docker run hello-world Quick test to confirm Docker can run a container.
Dockerfile needed A build command like docker build -t cheatsheet-site . only works if the current folder contains a valid Dockerfile.
FROM nginx:alpine Use nginx as a lightweight static site server.
COPY index.html /usr/share/nginx/html/index.html Copy your site file into nginx’s web root.
EXPOSE 80 Document that the container serves traffic on port 80.
Docker — Commands
docker build -t cheatsheet-site . Build an image from the current folder.
docker run -d -p 8081:80 --name cheatsheet-container cheatsheet-site Run the container in the background and map localhost:8081 to container port 80.
docker ps See running containers.
docker ps -a See all containers including stopped ones.
docker stop cheatsheet-container Stop the running container.
docker start cheatsheet-container Start the old container again. This does not rebuild your changes.
docker rm -f cheatsheet-container Delete the existing container, useful before recreating it.
Docker — Mental Model
Image = saved snapshot An image is the built version of your app at a moment in time.
Container = running copy A container runs from the image you built earlier.
docker start = reuse old version Starting a stopped container does not pull in new file changes.
Edit file → rebuild → rerun That is why Docker feels slower than npm for normal editing.
npm — Setup & Run
npm needs a package.json. Without that file, npm start will fail.
npm init -y Create a default package.json in the current folder.
npm install live-server --save-dev Install live-server as a dev dependency.
"start": "live-server" Add this inside the scripts section in package.json.
npm start Run the local dev server using the start script.
Ctrl + C Stop the running dev server in the terminal.
npx live-server Fast fallback if you want to run it without setting up a start script first.
npm — package.json Example
{ "scripts": { "start": "live-server" } } Minimal version needed for npm start to work for this site.
npm run See which scripts currently exist in package.json.
Missing script: "start" This error means the package exists, but the start script does not.
npm vs Docker
npm = local development Fast, simple, instant refresh. Best while building.
Docker = container testing Closer to a clean deployment environment. Best for verification.
Use npm while building Edit → save → see changes immediately.
Use Docker to verify Build and run a fresh container to make sure the app still works cleanly.
npm = speed Best for day-to-day editing.
Docker = consistency Best for testing how the app actually runs in isolation.

Tips to Remember
★ Quotes matter — folders with spaces need quotes, for example cd "Claude Docs".
★ The dot — . means current folder, so docker build -t app . and git clone URL . both use the current folder.
★ Push rejected? — pull first. Usually git pull origin main --rebase then git push.
★ Docker commands failing? — check Docker Desktop is open before debugging your files.
★ Missing package.json? — npm start will fail if the project has no package.json.
★ Missing script: start? — your package exists, but the scripts section needs "start": "live-server".
★ GitHub Pages pattern — sites usually live at https://delpedro.github.io/REPONAME/.
